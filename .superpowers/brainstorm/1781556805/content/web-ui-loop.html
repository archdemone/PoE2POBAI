Target "Option B": Frontend participates in the loop

Forwards: user msg → LLM; LLM's tool_calls → browser; browser's results → LLM
Server is a thin relay, browser owns the loop

Key Decision: Streaming Protocol

How does the browser talk to the server during the loop?

A

SSE (Server-Sent Events)

POST /api/chat returns a streaming response. Each event is a tool_call, tool_result, or text chunk. Browser renders as they arrive. Simple, one HTTP connection.

B

WebSocket

Persistent bidirectional connection. Browser sends messages, server streams back tool_calls, browser sends tool_results back, server continues. Most flexible, slightly more complex.

C

Repeated POST polling

Browser POSTs, server runs ONE tool_call iteration and returns. Browser POSTs result back, server continues. Loop unwound into individual request/response pairs. Simplest to implement, no new protocols.
