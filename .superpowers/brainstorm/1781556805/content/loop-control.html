Step 2 of 5: Tool Loop Control

Who owns the tool-call iteration? Server or browser?

Approach A: Server-driven (thin browser)

Server owns the loop, browser is a view layer
Browser → WebSocket: "is this build tanky?"
← Server streams progress events
"Running get_defenses... done."
"Running get_items... done."
← Final text response
↕
pobai-server (full loop + tool execution)
LLM call → tool_call → execute tool → result → LLM → repeat

Approach B: Frontend-driven (thick browser)

Browser owns the loop, server is a relay
Browser (owns loop logic)
1. Sends user msg via WS
2. Receives tool_calls from LLM
3. Executes tools itself (calls pob-mcp-server tools via HTTP API)
4. Sends results back via WS
5. Receives next chunk or tool_calls
6. Loop until done
↕
pobai-server (LLM proxy only)
Forwards: browser msg → LLM; LLM response → browser. Does NOT execute tools.
A

Server-driven

Server runs the loop + tools. Browser just renders. Faster to build — modify existing /api/chat. Less frontend complexity.

B

Frontend-driven

Browser owns the loop + executes tools via HTTP. Server is a thin LLM relay. More frontend work, but true "Option B" where the UI is the MCP client.
