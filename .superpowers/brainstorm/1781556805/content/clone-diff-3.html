Task #3 — Diff Output & Provenance

How are diffs presented? How do clones relate to originals?

Diff Output Format

A

Structured JSON diff

Machine-readable: {items: {added: [...], removed: [...], modified: [...]}, skills: {...}, passives: {...}}. Needed for MCP tool responses and programmatic use.

B

Human-readable summary

Natural language: "Swapped Ring 1 (Two-Stone Ring → Ming's Heart), added Herald of Ice, removed 3 passive nodes near Shadow." For chat display.

C

Side-by-side view (UI only)

Visual comparison table in the web UI. Not exposed via API/MCP. Purely a frontend rendering concern.

Clone Provenance

A

Linked clones (parent tracking)

Each clone stores parentId + patch that created it. Traceable lineage tree. "This variant was made by swapping ring + 5 nodes from build abc123."

B

Flat clones (standalone)

Clone is just a new independent snapshot. No provenance tracking. Simplest, but can't tell how a variant was made.

Implementation Location

A

In pob-mcp-server (MCP + store)

Add clone/diff/patch to the MCP server's snapshot-store. New MCP tools. HTTP server wraps them. Single source of truth.

B

In pobai-server (HTTP only)

Add clone/diff to the HTTP server, not MCP. Faster to build, but MCP-only users (Claude Desktop) can't use it unless MCP tools are also added.
